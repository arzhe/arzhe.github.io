CAEP文档

Posted on 2021-06-20 Edited on 2021-09-20 Views:

概述

  CAEP使用三个部分来构成策略权限决策模型：Applicability（适应性）、Condition（条件）和Effect（决策结果）。分别2个模块：PRM（策略请求模块）和PEM（策略实施模块）。PRM作为标准规范为用户与适配器（PEM）搭建桥梁，针对不同的策略语言，使用适配器来满足标准规范PRM。使用2个文件来分别描述这2个模块：prm.ini和pem.csv，亦或是其他文件（例如，db）。因此可以将适配器看作是“中间件”，CAEP默认其为csv读取库。prm.ini存储的是我们的访问控制模型，pem.csv存储的是我们具体的用户权限配置。

  权限的本质就是“某个用户，对某个资源，是否可以进行某些操作”，CAEP将此本质精练化，基本上只需要生成一个结构“Can”，构造这个结构的时候加载prm.ini和pem.csv。

prm.ini与pem.csv编写规则

ACL

  示例如下：

  如上所示，prm.ini支持#作为注释行标识符，每行为一个单独策略解释。对于ACL模型，prm.ini必须有4个区域：request_applicability、policy_applicability、condition和effect。方括号内的词汇称为section，区域内等号左边称为key，右边称为value。

  applicability区域内的表达式将分别匹配用户请求的格式与策略的格式，如果用户请求格式与request_applicability区域格式匹配，则将进行下一步动作，否则，返回一个错误。例如，用户发出的一个请求格式：Alice, file1, read适应于sub, res, act，则匹配成功。此后，CAEP将按policy_applicability区域的格式检索pem.csv文件中的策略语句，如果检索成功，则将策略存储在CAEP的存储区Model中以便与先前匹配成功的用户请求语句共同进行条件运算。否则，返回一个警告：提示用户没有编写相应的策略语句。

  如果applicability区域都成功匹配，并且CAEP的策略存储区Model非空，则CAEP将按照condition区域内的组合算法来计算出这些策略语句与用户请求语句的计算结果，其中计算结果有两种：permit和deny。

  如果上述过程顺利进行，则effect区域将进行最后的授权决策评估，该区域也是对“策略冲突”问题的一个解决方案。分为“允许优先”（permit_override）、“拒绝优先”（deny_override）和“顺序评估”（first_applicable）3种策略评估算法。顾名思义，“允许优先”算法中只要有一个策略语句匹配结果为true，则最后的决策评估为permit，否则为deny；“拒绝优先”算法中只要有一个策略语句匹配结果为false，则最后的决策评估为deny，否则为permit；“顺序评估”算法中策略语句匹配的顺序（也就是策略语句的编写顺序）将决定最后的决策评估结果，第一个匹配的策略语句如果为true，结果为permit，否则为deny。

解析prm.ini文件的库

  在CAEP中有config，rbac，effect和model一共4个库与解析prm.ini文件有关。

config库

  该库定义了一个config_interface.h的头文件，用来提供接口给用户，用户可以通过该接口来实现对prm.ini文件的读取操作。CAEP内置了prm.ini文件的读取操作：config.h，该接口提供了相应的函数来将prm.ini文件内的策略语句存储于CAEP配置存储区中，该存储区使用红黑树结构进行内存分配，将prm.ini中的每一个“区域”转换为三元组：section、key和value。并将其存储于unordered_map<string, unordered_map<string, string>> data中。

函数 返回值 作用 备注
ParseFile(string f_name) void 从传入参数的对应文件名的文件中解析“区域” private
ParseBuffer(istream* buffer) void 从传入参数的输入流中解析“区域” private
AddConfigData(string section, string key, string value) bool 将传入参数的三元组添加至存储区 private
NewConfig(string conf_name) shared_ptr<Config> 返回一个配置器，用户通过操作该配置器来对三元组数据进行处理 public static
NewConfigFromText(string text) shared_ptr<Config> 同上 public static
GetString(string key) string 传入的参数为section-key的一种字符串形式（以“::”分隔开来），返回对应section-key的字符串形式的value值 public
GetStrings(string key) vector<string> 同上，返回对应的section-key的以逗号分隔的vector形式的value值 public

rbac库

  该库为RBAC模型服务，该库定义了Role类，该库提供对角色的各类操作，并使用链式结构vector<Role*> roles来存储抽象数据“角色”，在CAEP中角色与用户都是以字符串形式存储的，因此角色与用户并没有严格的区分，但是用户可以在角色或用户前加上前缀来区分它们，例如：角色Apple与用户Apple，使用role::Apple以及user::Apple来区分。但是在CAEP中，将用户与角色统称为“角色”。Role的继承子角色（或用户）存储在vector容器中。RBAC库提供role_manager.h接口以实现角色管理功能，CAEP内置default_role_manager.h来进行角色的管理。

函数 返回值 作用 备注
NewRole(string name) Role* 以传入的参数name创建一个角色 public static
AddRole(Role* role) void 以传入的参数role添加一个继承子角色（或用户） public
DeleteRole(Role* role) void 以传入的参数role删除该继承子角色 public
HasRole(string name, int hierarchy) bool 判定在hierarchy继承层级下，是否有某角色 hierarchy为RBAC的继承体系的继承层次，public
HasDirectRole(string name) bool 判定RBAC体系的某角色是否有某直接继承其的子角色（一般用于判定某角色是否有某用户，相当于hierarchy为1的HasRole函数） 一般用于判定角色是否拥有某用户，public

  default_role_manager类实现了role_manager的接口，提供了unordered_map<string, Role*>类型的数据来管理CAEP所有的角色以及用户，该数据只负责角色的继承关系以及用户的依存信息的存储。
