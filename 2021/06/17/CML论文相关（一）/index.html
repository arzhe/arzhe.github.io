CML论文相关（一）

Posted on 2021-06-17 Edited on 2021-08-17 In 论文相关 Views:

摘要

  数据加密安全以及访问控制安全是海洋信息网络安全的重中之重，是在海洋信息系统中进行安全的信息管理的不可或缺的部分。但是众多访问控制机制以及安全策略语言导致了目前不同的系统有着自己的一套访问控制的实现体系，要想使不同的海洋信息网络系统之间自由沟通，便需要学习不同的访问控制策略，分别编写自己的安全策略；此外，各种晦涩难懂的，冗余语法结构的安全策略语言以及机制导致了使用者的学习与开发成本较高。本文提出了一种访问控制模型无关的、策略语言无关性、易于学习的安全策略语言SML以及实施机制SML-EM。用户可以较为轻松地在学习SML之后，实现出自己的安全策略系统，降低了学习以及开发的成本。SML支持BLP、RBAC、ABAC等访问控制模型，并且支持超级用户、多租用户和RESTFul等模块。

  海洋信息网络往往是军、民共用信息网络。海洋网络的数据即可用于海洋科研、海洋预报等，也可用于海洋战场环境保障、海洋目标监视与预警等军事目的。大量国际合作等海洋观测系统为人类认识、研究海洋提供了大量数据。但是，国家之间存在激烈的竞争，需要在确保海洋信息安全的同时，有利于国际合作，共享海洋数据。可见，急需展开海洋信息系统应用安全访问控制技术研究，以解决多部门，跨区域访问过程中的身份认证、授权管理和责任认定等访问控制安全需求。访问控制限制了用户可以直接做什么，以及代表用户执行的程序被允许做什么。通过这种方式，访问控制试图防止可能导致破坏安全性的活动[1]。访问控制作为提供信息安全保障的主要手段，及最突出的安全机制，被广泛地应用于防火墙、文件访问、VPN及物理安全等多个方面。近些年，有多种访问控制策略描述语言被提出，传统的有ASL（Authorization Specification Language）[2]和PDL（Policy Description Language）[3]，较新的有诸如XACML[4]、SPL[5]、Ponder[6]和Selinux[7]等。Jajodia等人提出的ASL是基于逻辑的授权定义语言，具有较强的计算能力，允许推理。但是它仅支持某一种访问控制模型，并且由于较差的可读性，导致不易于学习理解。PDL是基于事件的策略语言。PDL的基本格式为event-conditions-action，含义是在满足conditions的前提条件下，event的发生会触发action的执行。主要应用于基于策略的网络管理，不支持安全系统领域的访问控制技术。Carlos Ribeiro[5]等人提出了安全策略语言SPL（security policy language）。SPL可以同时表示多种复杂的授权策略，虽然足可以应对大多数复杂的权限策略情况，但是缺少自定义的权限策略的功能，在某些策略方面会收到约束。Ponder支持义务策略，即基于网络和分布式系统管理的策略的事件触发条件-动作规则。它是声明性的、强类型的和面向对象的，这使得它十分灵活、可扩展并适应广泛的管理需求[8]。然而其缺乏租户、算数运算符、逻辑运算符等表达特性，虽然支持静态策略冲突检测，但并没有相应的冲突消解机制。Selinux主要针对安全操作系统领域，支持多种安全模型，但是其缺乏有组织性的语法结构，因此对于每条策略规则都需要额外编写，这也就导致了采用Selinux的系统中的策略规则十分冗杂。

  当前海洋信息网络系统上存在多种不同的访问控制机制以及策略语言，由于它们不同平台上存在的较大差别，导致了实现海洋信息网络平台之间跨区域的信息交互访问控制系统较为困难。以下为它们之间的差异表现：

  1. 各平台系统采用的访问控制模型不同，例如有的只支持并采取的是ABAC模型，而有的仅支持RBAC模型，这使得开发，学习成本较大。

  2. 即使有的平台系统采用的是相同的访问控制模型，但是可能采取的策略语言不同，此外策略语言的语法复杂。开发人员仍然需要分别实现策略决策以及策略逻辑并需要付出较多的学习成本。

  目前多策略、混合策略成为访问控制策略领域的研究热点[9]，它们大多专注于提高策略的结构性以及逻辑性以提高策略描述的表达能力。这也导致了策略的描述语法越来越复杂，难以实现，并且学习成本越来越高。

  因此，本文提出一种访问策略模型CAEP-M（Categories Environment Policy Model），以下简称为CM。该模型使用CML（CAEP Model Language）策略描述语言来表达相应策略，并支持各类访问控制模型，如BLP（Bell-La Padula）[10]，RBAC[11]，ABAC[12]，ACL[13]等。在CM的基础上使用CML构建出策略实施机制CML-EM（SoPL enforcement mechanism）。CML语法较为简洁且直观，便于用户理解，也易于开发者学习。

CML语言

  CML适用于由IEFT[14][15]与ISO[16]定义的策略实施抽象模型。实施访问控制策略机制首先需要有合适的策略语言，然后针对特定的策略模型制定实施这些策略规则的逻辑。因此实施策略规则的逻辑一定程度上也反应了该策略语言的策略模型。一般来说，在一个支持混合策略、多用户、模型无关以及策略语言无关的系统中，需要允许用户自定义策略规则。此外，动态的策略实施逻辑在灵活性以及泛用性上要更优于静态的策略实施逻辑。目前主流的策略描述语言大多是静态的，在本文中，提出一种策略逻辑分离方案，将策略规则与策略逻辑进行分离。CML策略实施机制中实施的策略规则称为Policy，策略实施逻辑的抽象访问控制模型称为Model。更为通俗的来看，可用将Policy视为数据，Model视为逻辑。

策略模型

  CM通过两个组件来构成基本的抽象模型结构：策略请求组件（Policy Request Module，PRM），策略实施组件（Policy Enforcement Module，PEM）。PRM为权限决策制定一个请求，该请求包括对访问请求的描述：发出请求的主体、要请求访问的资源、要执行的操作以及其他的相关信息。然后该授权决策请求会被发送到PEM组件。PEM组件会检索与决策请求匹配的策略，以及评估这些策略所需的任何额外附加信息，并向PRM返回授权决策。授权决策可以是“allow access”、“deny access”或是“no policy apply to this request”。如果PRM由于某种原因无法检索策略以及评估所需的信息，则PRM会返回一个错误而不是授权决策。根据从PRM发送而来的授权决策或错误，PEM将会授予、拒绝访问请求或是以某种方式回应错误。

  PRM无法控制策略决策的执行，这是PEM的功能。PEM通过在应用程序或平台上的具体实现来实现“拦截和描述访问请求”的功能。PRM以及其使用的CML是独立于应用程序和平台的。CM通过将策略的请求和实施进行分离来完成授权决策系统的实现，PEM具体在代码层面，PRM具体在策略语言（CML）层面。

授权决策请求

  CML为授权决策请求指定了一种抽象格式，以描述试图进行资源访问的属性。这些属性是在描述策略时用来引用的某类型的值。该策略属性将在后文中介绍。以下为一个例子：

NAME TYPE VALUE
role string “administrator”
resource URL（string） “http://www.arzhe.com"
action string “read”

  在该例中，一个名为“administrator”的角色试图读取一个URL地址。反过来说，要想读取URL的值，用户的角色属性必须是“administrator”的值，只有这样，访问才会被允许。

策略属性

  在CML中，每个策略属性都有唯一的名称，并且类型值都是字符串。其中有两种类型的策略属性：类别（categories）和环境（environment）。categories属性有三种类别：Subject、Resource和Action（主体、资源和操作）。Subject类别描述与发出访问请求相关联的主体；Resource类别描述要访问的资源；Action类别描述要对资源执行的操作。

  Environment属性描述与Subject、Resource和Action不相关的访问请求的特征。例如进行访问的时间，或是进行访问的主体的年龄。 尽管环境（Environment）属性可以与类别（categories）属性描述相同的实体，但是环境属性与任何类别属性之间没有任何显式的关联。例如，一个名为age的环境属性可能与用户的年龄或资源的年龄相关联，但是这种关联SML本身不会做出任何的定义，而是需要用户去自定义。

  策略规则中的条件可以引用策略属性的名称，例如role，resource，action等，并且该属性值必须可用才能对条件进行计算，如果在计算条件时，所需的策略属性名称没有合理的值或者是空值，则策略计算将返回错误或是不作出任何响应。

策略词汇

  策略词汇可以帮助应用程序编写人员和策略编写人员弄清楚必须提供哪些属性。但是，以这种方式指定词汇并不能确保在评估策略时所有必要的属性都实际可用。需要注意的是，词汇规范在CML中并不是最重要的概念，但可以通过定义一个属性来包含其他地方指定的属性集合标识，无需自己来扩展语言也可以支持它。然后，策略、子策略或规则可以要求PEM为这个词汇属性提供值。策略可以要求这些值包括计算策略相关部分所需的属性集合的标识。如果这些都不匹配，将返回一个错误。此外，可以编写属性检索组件来解释这种分层关系的外部规范，以便在请求时返回某些指定属性值的父系或子系。

  将策略词汇指定为策略的一部分，在实际作业中可能难以实现。例如，策略可能要求必须提供一个“Age”属性或一个“Private Key”属性。如果词汇在同一个策略属性中列出了这两个属性，那么只支持其中一种属性的用户将永远无法获得访问，因为同一个策略属性不能够被同时两个属性赋值。

  鉴于上述情况，只能要求用户足够“聪明”，能够编写反馈出上述错误的策略语句，尽管如此，内置的CML机制还是会帮助用户来检测出错误，并返回错误代码。

授权决策请求

  在CML中，决策请求必须包含一个或多个用户类别属性、数据类别属性、操作类别属性和目的类别属性，以及各种容器属性集的可选值。如果授权决策请求中有多个用户类别属性，它们都必须引用相同的主体，尽管单个决策请求可能包含多个独立的数据类别、操作和用途。

  CML决策请求由Subject属性、Resource属性、Action属性和Environment属性以任意组合方式构成。在CML核心规范中，一个请求可能只包含一个Action，但可能包含多个资源或主体。每个Subject都是以某种方式进行访问请求的独立主体，并且每个Subject都有自己的Subject属性集合。默认的主体是进行访问请求的主体，主体可以是任意的。在策略中，访问可以以这些Subject属性的任何组合为条件。在CML中，可以将其他主体指定为环境属性，但是这些附加主体并不支持CML内置的功能，需要用户额外编写规则适应条件，所幸的是，CML提供了接口规范以供需要该功能的用户来借鉴。

授权决策响应

  在CML中，总是会返回“一个”响应，其中包含“Allow”、“Deny”或“no policy apply to this request”的结果，或者如果在策略评估期间发生错误，则返回错误状态。响应包含与应用于用户类别、数据类别、操作和目的的每个组合的规则相关的标识符。

策略评估

  在CML中，除非是指定了多个resource，否则将针对授权决策请求中的所有属性对策略进行一次性评估，并将评估结果作为授权决策返回。如果指定了多个resource，授权决策请求将被分解为多个简单请求，每个简单请求仅包含一个resource，每个简单请求将分别评估。如果授权决策请求请求一个总体决策，那么授权决策将仅在评估所有简单请求的结果表明允许访问时才表示允许访问。如果任何一个结果表明拒绝访问，那么将作为总体决策返回；否则，将返回指示不应用任何策略的决策。CML会为多个resource返回单个决策。此外，可以使用单个授权决策请求返回这些单独的决策。做法就是使用通配符函数，这需要用户使用CML内置的Matching函数并根据策略规范来进行配置。

嵌套策略

  对于管理复杂的策略，允许一个策略由多个子策略组成是非常有用的，这些子策略被分别进行评估、编写和维护。例如，“子策略”允许不同的模块管理自己的策略，同时确保在达成授权决策时考虑到所有相关的策略。即使在集中管理策略的地方，使用“子策略”也允许将策略分组为更小的单元，从而更容易、更有效地维护它们，并选择适用于特定类型请求的策略。

  CML并不支持嵌套策略的，这样做的原因是为了极简化CML的语法规则，避免使用户学习成本过大。取而代之的是，CML使用“策略集”：一个策略集包含一个或多个子策略，这些子策略可以包含其他策略集。每个策略集和子策略都包含自己的条件评估算法，只有满足其条件评估算法时，才对任何特定的子策略树进行评估。子策略可以内联声明，也可以使用策略标识符引用。策略引用允许使用分布式策略，支持分散的策略管理。

决策结果冲突

  由于CML的策略可能包含多个规则，而且由于对同一请求进行评估时，不同的规则可能返回不同的结果，因此在规则冲突时，必须有一种方法确定从策略评估返回的授权决策。此外，由于CML策略可能还包含多个子策略，而且这些子策略也可能得到不同的结果，因此CML中还必须有一种方法来解决子策略冲突。

一种方法是通过按顺序计算规则，直到其中某一个规则适用于给定的授权决策请求，从而避免了冲突。对该规则的计算结果将作为授权决策返回。此外，在CML中，每个策略都包含一个规则组合算法的标识符，该算法将用于组合特定策略中的规则结果。CML规范还定义了几个标准规则组合算法，包括deny-overrides（如果任何规则计算为deny，则返回deny）和permit-overrides（如果至少有一个规则计算为permit，则返回permit）。每个CML策略集同样包含策略组合算法的标识符，用于组合该策略集中的子策略的结果。标准的策略包括deny-overrides、permit-overrides和first-applicable（返回对应用于请求的第一个策略进行评估的结果）。
