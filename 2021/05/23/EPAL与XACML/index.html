EPAL与XACML

Posted on 2021-05-23 Edited on 2021-06-14 In 论文 Views:

介绍

  用于支持权限策略的表达和实施的标准结构化语言必须满足以下几个要求：

  1. 为了支持权限策略的表达，该语言不能够仅仅只在“谁对哪些资源可以执行哪些操作”上进行约束，还必须要将数据收集的目的与访问的目的相匹配。
  2. 为了支持自动化执行，该语言必须支持可直接执行的策略表达，策略使用标识符来描述主体、资源、动作和访问条件，而这些所谓的标识符可以直接自动化地映射到计算机程序所使用的实际对象和操作。
  3. 该语言必须是平台无关性的，以便将其作为标准使用。

  有两种平台无关性的语言满足上述大部分的要求，一个是IBM’s Enterprise Privacy Authorization Language(EPAL)，另外一个是OASIS eXtensible Access Control Markup Language(XACML)。

  对于企业来说，如果要选择最适合他们的需求的语言，了解这两种语言之间的差异是十分重要的；对于研究人员来说，如果要研究出比这两种语言更优秀的语言，了解这两种语言之间的差异亦是十分重要的。而后者则正是我写下此文的目的。

EPAL和XACML概述

抽象模型

  EPAL和XACML都使用了由IETF与ISO定义的策略实施抽象模型，在该模型中，所有访问受保护的资源的请求都要经过一个称为策略实施点（Policy Enforcement Point）的抽象组件或是PEP（Access Enforcement Function in the ISO standard）。PEP为授权决策制定了一个请求，该请求包括对访问请求的描述：谁发出请求、请求什么资源、要执行什么操作、访问请求的目的以及其他的相关信息。该授权决策请求被发送到另一个称为策略决策点（Policy Decision Point）的抽象组件或是PDP（Access Decision Function in the ISO standard）。PDP会检索与决策请求匹配的策略，以及评估这些策略所需的任何附加信息，并向PEP返回授权决策。在EPAL和XACML中，授权决策可以是“allow access”，“deny access”或是“no policies apply to this request”。如果PDP由于某种原因无法检索策略或评估所需的信息，则PDP可能返回一个错误而不是授权决策。根据授权决策，PEP授予或拒绝访问请求。一个授权决策可能伴随着义务，这些义务是PEP在实施授权决策时必须采取的动作。所谓的义务，一个典型的例子就是记录访问请求。

  PDP无法控制策略决策的执行，这是PEP的功能。PEP的实现通常是特定于应用程序或平台的，因为拦截和描述访问请求通常需要特定于应用程序或平台的实现细节。然而PDP及其使用的策略语言可以是独立于应用程序和平台的。

授权决策请求

  EPAL和XACML都为授权决策请求指定了一种抽象格式，以描述试图进行资源访问的属性。这些是在说明策略时可引用的命名的类型化值。以下是一个例子：

NAME TYPE VALUE
role string “administrator”
resource-id URL “http://www.example.com/acount/1234"
action-id string “read”
purpose string “verify balance”

  在该例中，一个名为“administrator”的角色试图读取一个URL地址来确认账户余额。如果策略反过来，为了读取URL resource-id，用户的角色属性必须具是“administrator”的值，而目的属性必须是“verify balance”，只有这样，访问才会被允许。

  在这两种语言中，由PDP评估的授权决策请求与从PEP发送的任何特定消息格式的语法或内容之间只有弱关系；虽然PEP可以使用抽象格式作为其原生请求格式，但也可以将各种特定于应用程序的PEP请求格式映射到抽象格式。

策略结构

  在EPAL和XACML中，一个策略由一个或多个规则组成，这些规则每一个都有3个主要部分，如下：

Part Description
适用性（Applicability） 如果满足某条件，则规则适用于一个请求
条件（Condition） 如果满足某条件，则规则满足
效果（Effect） 如果规则同时满足与适用，则规则返回{permit,deny}

  Applicability是一组关于决策请求中的属性值的简单约束，这些约束决定规则是否适用于给定请求。Condition是一个称为条件的谓词，该条件由拥有请求（或从其他来源获得）中的字面值和属性值的函数组成。Effect则是一个效果（在EPAL中称为结果），如果规则适用并且满足，它将被返回{permit,deny}。一个Effect可以指示为“permit acces”或者是“deny acces”。可选的第四个组件是Obligations（义务），它是一个将与授权决策一起返回给PEP的命名值。隐私策略中一个典型义务是访问的资源不能与第三方共享，并且接收方必须在30天内销毁信息。对于PDP来说，义务是不透明的，理解并履行任何义务的责任落在了PEP身上。

策略评估

  首先通过确定适用性是否是被满足来评估规则。如果不适用，则不执行进一步的计算，并返回一个指示不适用的值。如果满足适用性部分，则对条件部分进行评估。如果结果为true，则返回规则的效果（Effect），如果结果不为true，则返回一个指示不适用的值。如果在评估适用性或条件部分时出现任何错误，则可能返回一个错误值。EPAL和XACML在如何确定策略评估的总体结果方面有所不同，因为不同的规则可能返回不同的值。它们在错误和义务的处理方式上也有所不同。这些差异将在下文一一展示。

比较

  虽然EPAL和XACML在结构和概念上非常相似，但两种语言之间的差异是显著的。并且极大地影响了它们的可用性和满足Web服务隐私策略语言需求的能力。以下将讨论一些关键性差异。

策略属性

  EPAL和XACML在策略变量（或属性）词汇表结构上有所不同。在EPAL中，有两种类型的属性：类别（categories）和容器属性（container attributes）。categories有四种类别：用户类别、数据类别、操作和目的。categories属性是与用户组、资源组、操作组和目的组相关联的名称，从隐私角度来看，这些组的处理方式是不同的。例如，属于具有adult值的用户类别的用户与属于具有child值的用户类别的用户可能会受到不同程度的对待。用户类别值与发出访问请求的主体相关联。数据类别，操作和目的的类别值与被访问的资源，要在资源上执行的操作以及访问资源的目的相关联。最后要注意的是，类别属性值都是字符串。

  EPAL容器属性（container attributes）用于保存策略评估所需的任何不属于隐私类别的信息。尽管容器属性可以与类别属性中的某一类别描述相同的实体，但是容器属性与任何类别类型或值之间没有显式的关联。例如，一个名为age的容器属性可能与用户的年龄或资源的年龄相关联，但语言本身不会做出任何这样的关联。每个容器属性都有一个唯一的名称和一个类型化值；支持的类型包括字符串、整数、布尔值、日期、时间等。容器属性被分组到称为容器的命名集中，容器属性的名称则来自容器。规则中的条件可以引用特定容器的属性名称，并且该属性值必须可用才能对条件进行计算：如果在计算条件时所需的容器没有被完全填充，则策略计算将返回一个错误。

  在XACML中仅有一种类型的属性，每一个属性都有唯一的名称和类型化值。属性的名称可以是URN或XPath表达式。每一个XACML属性都与四个类中的一个相关联：Subject、Resource、Action和Environment（主体、资源、操作和环境）。Subject属性描述与发出访问请求相关联的主体。Resource属性描述要访问的资源。Action属性描述要对资源执行的操作。Environment属性描述与Subject、Resource或Action不相关的访问请求的特征，例如进行访问的时间或需要进行访问的交易所依据的合同标识符。

  EPAL的容器属性与XACML的环境属性不一样。容器属性是除与隐私相关的类别以外的任何属性；它可以描述访问用户的特征、被访问的资源、操作或目的。事实上，用户、资源、操作或目的的任何属性，除了它们的隐私类别之外，一定都是容器属性。XACML的环境属性则是那些不适用于主体、资源或操作的属性。

XML属性值

  XACML支持使用XPath表达式作为命名属性的一个可选项。通常通过一个被请求访问的XML文档副本，或者通过一个值为XML元素的请求属性对XPath表达式求值来计算出属性值。例如，XACML中允许有这样的条件判定式：在请求中的“subject-id”属性的值必须与“//patientRecord/patientName/text()”节点或“//patientRecord/PhisicianName/text()”节点的值相等。因此，XACML中的通用检索组件将会把XML文档中的节点作为属性处理。

  EPAL没有“使用被访问资源的内容来指定属性值”的内置机制；它不支持具有作为XML模式元素实例的值的属性，也不支持在被访问的XML文档中使用节点作为属性。如果策略要求XML资源的一个节点必须匹配给定的值，那么EPAL词汇表必须将该特定节点定义为具有自己名称的单独属性。此外，必须额外编写属性检索组件来处理对每个这样的属性值的检索和映射。这也是EPAL在为Web服务应用程序提供支持方面的一个主要缺陷。

策略词汇

  除了策略属性上的差异外，这两种语言在处理总体策略词汇方面也有所不同。每个EPAL策略都包含一个对抽象结构的引用，这个抽象结构称为“词汇”，它描述了可以在该策略中使用的所有属性、容器、条件和义务。策略本身是使用对词汇结构中定义的属性、条件和义务的引用来表示的。相比之下，在XACML中，属性、条件和义务可以直接用于表达策略，也可以被定义为变量并在表达策略时来引用他；此外，也并不需要将所有变量定义在一个位置。

  然而，在同一个位置定义策略元素也有某些好处。最主要的一个就是可以定义属性之间的关系。EPAL以两种方式来使用该功能：

  1. EPAL允许将类别属性定义为层次结构，其中类别属性可以指向另一个类别属性作为其父类。
  2. 词汇结构允许EPAL使用容器，容器可以用于对来自一个源的属性进行分组，从而帮助优化属性检索。

  词汇还可以帮助应用程序编写人员和策略编写人员就必须提供哪些属性达成协议。但是，以这种方式指定词汇并不能确保在评估策略时所有必要的属性都实际可用。需要注意的是，词汇规范在XACML中并不是最重要的概念，但通过定义一个属性来包含其他地方指定的属性集合标识，无需扩展语言也可以支持它。然后，策略、子策略或规则可以要求PEP为这个词汇属性提供值。策略可以要求这些值包括计算策略相关部分所需的属性集合的标识。如果这些都不匹配，将返回一个错误。这种词汇属性的值引用的规范还可以定义属性之间的层次关系。可以编写属性检索组件来解释这种层次关系的外部规范，以便在请求时返回某些指定属性值的父系或子系。

  将策略词汇指定为策略的一部分（比如在EPAL中），在实际作业中可能过于苛刻。例如，策略可能要求必须提供一个“Challenge Response”属性或一个“Private Key Authenticator”属性。如果词汇在同一个容器中列出了这两个属性，那么只支持其中一种机制的用户将永远无法获得访问，因为容器永远不会被完全填充。

  将特定格式的词汇指定为策略的一部分还存在其他问题。由于EPAL是通过引用将词汇包含在内的，因此多个策略可能包含相同的词汇。对一个策略所需的词汇的更改可能会导致使用同样词汇的另一个策略结构出现问题。这使得实现一致性策略更加困难。另一个问题是，策略和各种定义属性的外部规范之间的关系要么是间接的（通过描述性注释），要么是丢失的。但是，由于XACML中的词汇属性可以有多个值，因此可以包含正在使用的实际外部规范的标识，并且可以编写属性检索组件将这些规范中定义的属性转换为XACML的抽象属性格式。

授权决策请求

  在EPAL中，决策请求必须包含一个或多个用户类别属性、数据类别属性、操作类别属性和目的类别属性，以及各种容器属性集的可选值。如果授权决策请求中有多个用户类别属性，它们都必须引用相同的主体，尽管单个决策请求可能包含多个独立的数据类别、操作和用途。请求的语义是“我属于这些用户类别”。是否至少有一个用户类别可以对所有这些数据类别执行所有这些操作？

  XACML决策请求由Subject属性、Resource属性、Action属性和Environment属性的任意集合组合而成。在XACML核心规范中，一个请求可能只包含一个Action，但可能包含多个资源或主体。每个Subject都是以某种方式进行访问请求的独立主体，并且每个Subject都有自己的Subject属性集合。默认的主体是进行访问请求的主体，主体可以是任意的。在策略中，访问可以以这些Subject属性的任何组合为条件。在EPAL中，可以将其他主体指定为容器属性，但是这些附加主体不具有EPAL与用户类别相关联的特殊策略评估语义，如下文所述。

授权决策响应

  在EPAL中，总是会返回单个响应，其中包含指示“允许访问”、“拒绝访问”或“没有策略匹配此访问请求”的结果，或者如果在策略评估期间发生错误，则返回错误状态。响应包含与应用于用户类别、数据类别、操作和目的的每个组合的规则相关的标识符，以及与这些规则相关的义务。

  在XACML中，单个授权决策请求可以请求包含一个总体决策的响应，或者请求中指定的每个资源的单独决策。每个决策都表明允许访问、拒绝访问、没有策略适用于此访问请求，或者由于一个错误“无法确定应该允许访问还是拒绝访问”。每个决策包括与该决策相关的资源的名称，以及与该资源的决策相关联的任何义务。如果在处理过程中发生错误，决策将由策略中指定的组合算法确定（如下所述）。错误通常只有在PDP无法确定应该或者不应该允许访问时才终止评估。如果请求访问XML文档中的多个节点，则每个节点的决策都包含一个XPath表达式。

策略评估

  以下是策略评估语义的相关概念，在这两种语言中，都可以以任何保留整体语义的方式自由实现求值算法。

  一个EPAL授权决策请求被分解为一个或多个简单请求，每个请求都包含用户类别、数据类别、操作和目的的属性值的单个组合；在评估这些简单请求时使用同一组容器属性，因此容器属性必须平等地应用于决策请求中的所有用户类别、数据类别、操作和目的。简单请求的计算顺序是在策略词汇中指定用户类别值的顺序。如果在任何策略评估过程中发生错误，将立即返回一个错误作为授权决策响应。否则，如果包含任何单个用户类别值的所有简单请求的评估返回指示允许访问的结果，那么评估将终止，并返回指示允许访问的授权决策。此外，如果对任何单个简单请求的评估产生指示拒绝访问的结果，则返回指示拒绝访问的结果作为授权决策响应。

  在XACML中，除非是指定了多个resource，否则将针对授权决策请求中的所有属性对策略进行一次评估，并将评估结果作为授权决策返回。如果指定了多个resource，授权决策请求将被分解为多个简单请求，每个简单请求仅包含一个resource，每个简单请求将分别评估。如果授权决策请求请求一个总体决策，那么授权决策将仅在评估所有简单请求的结果表明允许访问时才表示允许访问。如果任何一个结果表明拒绝访问，那么将作为总体决策返回；否则，将返回指示不应用任何策略的决策。XACML会为多个resource返回单个决策，同样，EPAL也会为多个数据类别返回单个决策。但是，在EPAL中，为了使每个数据类别分别获得单独的决策，必须为每个数据类别提出单独的请求。而在XACML中，可以使用单个授权决策请求返回这些单独的决策。

  EPAL语义不支持要求请求者是多个用户类别成员的策略。例如，如果医疗保健提供者和医院工作人员是单独的用户类别，则EPAL策略不可能要求请求者同时属于两个用户类别；如果允许任一用户类别访问，则允许访问。

规则

  EPAL规则的适应性部分不能引用容器属性，条件部分不能引用类别属性。在XACML中，任何属性都可以在规则的任何部分中使用。在EPAL中，适应性谓词都与请求中必须出现的类别值完全匹配。在XACML中，任何返回布尔结果并将属性值与文字值进行比较的函数都可以在适应性谓词中使用。由于规则的适应性部分用于索引规则或优化规则评估，因此EPAL限制可能会限制索引或优化。EPAL和XACML支持一组广泛的标准函数，可用于说明访问条件；这两种语言支持的函数几乎是相同的。

嵌套策略

  对于管理复杂的策略，允许一个策略由多个子策略组成是非常有用的，这些子策略被分别进行评估、编写和维护。例如，“子策略”允许不同的模块管理自己的策略，同时确保在达成授权决策时考虑到所有相关的策略。即使在集中管理策略的地方，使用“子策略”也允许将策略分组为更小的单元，从而更容易、更有效地维护它们，并选择适用于特定类型请求的策略。

  EPAL不支持嵌套策略，每个策略都是独立的，没有语言定义的机制来组合应用于给定请求的多个策略的结果。XACML不支持嵌套策略：一个策略集包含一个或多个子策略，这些子策略可以包含其他策略集。每个策略集和子策略都包含自己的适用性谓词，只有满足其适用性谓词时，才对任何特定的子策略树进行评估。子策略可以内联声明，也可以使用策略标识符引用。策略引用允许使用分布式策略，支持分散的策略管理。

决策结果冲突

  由于XACML和EPAL中的策略可能包含多个规则，而且由于对同一请求进行评估时，不同的规则可能返回不同的结果，因此在规则冲突时，必须有一种方法确定从策略评估返回的授权决策。此外，由于XACML策略可能还包含多个子策略，而且这些子策略也可能得到不同的结果，因此XACML中还必须有一种方法来解决子策略冲突。

  在EPAL中，通过按顺序计算规则，直到其中某一个规则适用于给定的授权决策请求，从而避免了冲突。对该规则的计算结果将作为授权决策返回。在XACML中，每个策略都包含一个规则组合算法的标识符，该算法将用于组合特定策略中的规则结果。XACML规范定义了几个标准规则组合算法，包括deny-overrides（如果任何规则计算为deny，则返回deny）和permit-overrides（如果至少有一个规则计算为permit，则返回permit）。每个XACML策略集同样包含策略组合算法的标识符，用于组合该策略集中的子策略的结果。标准的策略包括deny-overrides、permit-overrides和first-applicable（返回对应用于请求的第一个策略进行评估的结果），这相当于EPAL的一个冲突解决机制。XACML可以自由定义和使用其他组合算法。

继承角色和类别

  正如ANSI基于角色的访问控制标准所定义的，角色是权限的集合。它旨在对组织中执行某些功能角色所需的权限进行建模。角色不同于EPAL类别，因为类别是实体的集合，而不是权限的集合。
